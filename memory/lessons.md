# memory/lessons.md

> 教训层：记录踩坑与经验，按严重程度分级。

## 严重程度说明
- **P0**: 严重故障 / 数据风险 / 安全风险
- **P1**: 影响主要功能或用户体验
- **P2**: 一般问题，可接受但需优化
- **P3**: 轻微问题 / 习惯改进

## 模板

### [P1] 标题
- **日期**: YYYY-MM-DD
- **现象**: 发生了什么
- **根因**: 为什么发生
- **修复**: 做了什么
- **预防**: 如何避免再次发生
- **标签**: #tag1 #tag2

## 已记录

### [P1] 长任务期间未及时回执，导致用户感知“卡住”
- **日期**: 2026-03-05
- **现象**: 执行重启/索引等长任务时，未先回复“收到，处理中”，用户需要补发“？”才收到反馈。
- **根因**: 长阻塞调用未做分段回执与轮询汇报。
- **修复**: 在 AGENTS.md 增加响应时效规则（先回执、10秒汇报、短超时轮询、超时降级）。
- **预防**: 所有 >10 秒任务必须先回执并持续状态更新。
- **标签**: #response-latency #ux #process

### [P2] 低 RPM 配额下不应启用并发子 agent 方案
- **日期**: 2026-03-05
- **现象**: 当前模型额度为每分钟 4 次，请求密集时并发方案容易触发排队/超时风险。
- **根因**: 方案设计未充分匹配实际吞吐上限。
- **修复**: 当日决策暂缓启用子 agent 并发流程，保留串行执行。
- **预防**: 新流程上线前先校验速率配额与并发模型，超限场景默认降级为串行。
- **标签**: #rate-limit #capacity-planning #subagent
